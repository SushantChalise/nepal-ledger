# ADR-0009: Source Registry — single source of truth

- **Status:** Accepted
- **Date:** 2026-05-14
- **Deciders:** Mother Opus; user ratified
- **Tags:** doctrine, source-registry, data-strategy, ci

## Context

The `source_registry` table has been the live truth since migration 0001, but the human-readable surface was split across three drifting artifacts:

- `docs/SOURCE_REGISTRY.md` enumerated 12 entries in 4 tiers (the Day-0 list).
- `docs/SOURCE_REGISTRY_AUDIT_PROPOSAL.md` proposed ~40 entries in 5 tiers + reference assets, still marked "Draft for user review".
- `scripts/seed-source-registry.ts` seeded only the original 2 starter rows.

[`docs/DATA_REQUIREMENTS.md`](../DATA_REQUIREMENTS.md) §6 OQ-4 surfaced the resulting ambiguity: which of the three artifacts is the source of truth? ADR-0008 reserved this ADR-0009 slot for the answer.

The cost of leaving the question open is real. Each parser brief (NRB BFI monthly XLSX, MoF Red Book, intergovernmental fiscal transfers, etc.) must cite a registered source before its parser ships ([SOURCE_REGISTRY.md §"Workflow"](../SOURCE_REGISTRY.md)). Briefs were silently consulting the audit proposal as if it were canonical; the registry doc lagged behind reality.

## Decision

The `source_registry` table is the canonical source of truth. The seed script is the declarative form. Markdown documents downstream of either are generated or cite-only.

Concretely:

1. **`scripts/seed-source-registry.ts` is the canonical declarative definition.** Its `ROWS` array is the lifetime source for what the project intends to register. The script is idempotent (composes `upsertSource`, keyed on `source_id`) and dry-runnable without `DATABASE_URL`.
2. **`docs/sources/_index.md` is auto-generated and committed.** Generated by `pnpm gen:source-index` from the same `ROWS`. Hand-edits are rejected by the CI gate. Committed (not generated-on-CI-and-thrown-away) so PR reviewers see the diff in the registered set.
3. **`docs/sources/<source-id>.md` is the per-source profile.** Required for every non-`reference_only` row. Stubs are acceptable at registration time; rich profiles (breakage modes, revision policy, indicator list) land with the parser PR.
4. **`docs/SOURCE_REGISTRY.md` retains the schema + workflow + Markdown template.** It no longer enumerates the registered set; it points at `docs/sources/_index.md` and at the `source_registry` table.
5. **Tier-0–4 semantics are locked** at the column level (`tier smallint`):
   - `0` = pre-ADR-0009 baseline (the two starter rows registered in migration 0001).
   - `1`–`4` = phased rollout per BACKEND_PLAN (Tier 1: Days 1–28, Tier 4: Days 120–365).
   - `null` = reference-only assets (we cite but don't parse into `approved_indicator_values`), or rows registered before the column landed.
   - Changing tier semantics or adding tier 5+ requires a new ADR.
6. **`ingestion_mode` enum is locked** to `automated_cron | manual_upload | reference_only`. New values require a new ADR.
7. **`status` lifecycle is locked** to `active | paused | deprecated`. Only `active` feeds the `approved_indicator_values` promotion pipeline.
8. **Adding a source = regular PR.** No ADR. The PR adds the row to `seed-source-registry.ts`, regenerates `_index.md`, adds (or stubs) the per-source profile, and bumps `last_verified_at` on touched rows. CI enforces all three artifacts.
9. **Schema/enum/tier-definition changes = ADR required.** New columns, new enum values, new tier numbers, or changes to the workflow itself need a fresh ADR that supersedes this one or amends it explicitly.

This ADR supersedes the implicit convention that `SOURCE_REGISTRY.md` was the enumeration.

## Alternatives Considered

### Option A: Keep `SOURCE_REGISTRY.md` as the prose enumeration; seed script trails it
- **Pro:** No tooling work; humans edit one file.
- **Con:** Prose enumerations drift faster than data. The audit proposal already drifted from `SOURCE_REGISTRY.md` within Day 0, and the seed drifted from both.
- Rejected.

### Option B: Generate `SOURCE_REGISTRY.md` entirely from the seed
- **Pro:** Strongest convergence guarantee.
- **Con:** Generating the schema/workflow/template prose alongside the registered set forces a mixed-template generator. Workflow is rarely-changed prose; the registered set changes weekly. Two artifacts (schema doc + generated index) cleaner.
- Rejected.

### Option C (chosen): Seed is canonical; index is generated; schema doc carries the template and points at the index
- **Pro:** Separates rarely-changed prose (workflow + template) from frequently-changed data (registered set). CI-enforceable.
- **Con:** Two new scripts (gen + check). One-time setup cost.
- Accepted.

## Consequences

### Positive

- One canonical surface (`scripts/seed-source-registry.ts`) for every source the project tracks; no more drift across three artifacts.
- PR reviewers see registry additions/removals as a diff in both the seed AND `_index.md`; the second view catches subtle changes (e.g. tier shifts, ingestion-mode flips).
- New sources land via plain PR — Worker briefs can register a source without invoking the ADR ceremony, removing a long-standing friction point.
- The `tier` column makes the BACKEND_PLAN phasing queryable from the live database (`SELECT * FROM source_registry WHERE tier = 1 AND status = 'active'`).
- CI gate forces the contract: anyone bypassing the seed (e.g. raw INSERT in a migration) is caught when their `_index.md` diverges or their profile is missing.

### Negative

- `pnpm install` now pulls `tsx` as a devDep so the generator/check scripts run cross-platform. Net cost: one transitive dep, ~5 MB.
- The generator/check scripts are new code surface (~120 lines combined) that must stay maintained. Mitigated by being small and pure (no DB, no network).
- Reference-only rows now live in `source_registry` alongside parsed feeds. Tooling that joins `source_registry → parser_runs` must filter `ingestion_mode != 'reference_only'`. Documented in the seed file's tier comment.

### Neutral

- Existing migrations 0001 + 0002 are unchanged. Migration 0003 adds the nullable `tier` column; backfilling for the two starter rows happens via the next `pnpm exec tsx scripts/seed-source-registry.ts` run (idempotent).
